## Express (Node.js)

The traditional approach: run MultiTenantKit on your own Express server with full control over your infrastructure.

### 1. Setup Project

```bash
npm init -y
npm install @multitenantkit/sdk dotenv
npm install -D typescript tsx @types/node
```

### 2. Environment Variables

Create `.env`:

```bash
DATABASE_URL=postgresql://user:password@localhost:5432/postgres
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_SERVICE_KEY=your-service-role-key
```

### 3. Create Server

Create `src/index.ts`:

```typescript
import 'dotenv/config';
import { createSupabaseExpressApp } from '@multitenantkit/sdk';
import { z } from 'zod';

// Define custom fields (optional)
const customUserFields = z.object({
  firstName: z.string().min(1).max(100),
  lastName: z.string().min(1).max(100),
});

const customOrgFields = z.object({
  organizationName: z.string().min(1).max(200),
  industry: z.string().optional(),
});

// Create app with custom fields
const app = createSupabaseExpressApp({
  users: {
    customFields: { customSchema: customUserFields }
  },
  organizations: {
    customFields: { customSchema: customOrgFields }
  }
});

const port = process.env.PORT || 3000;
app.listen(port, () => {
  console.log(`ðŸš€ Server running on http://localhost:${port}`);
  console.log('');
  console.log('Endpoints:');
  console.log('  GET    /users/me');
  console.log('  PATCH  /users/me');
  console.log('  POST   /organizations');
  console.log('  GET    /organizations/:id');
  console.log('  ...and 14 more!');
});
```

### 4. Run

```bash
npx tsx src/index.ts
```

### Alternative: Add to Existing App

Already have an Express app? Use `createExpressRouter`:

```typescript
import 'dotenv/config';
import express from 'express';
import { createExpressRouter } from '@multitenantkit/sdk';

const app = express();

// Your existing routes
app.get('/api/billing', (req, res) => res.json({ status: 'ok' }));

// Add MultiTenantKit under a prefix
const mtRouter = createExpressRouter();
app.use('/api/teams', mtRouter);

app.listen(3000);
```

## Supabase Edge Functions

Deploy MultiTenantKit as a serverless "fat function" on Supabase's globally distributed edge network.

### 1. Create Edge Function

```bash
# Initialize Supabase (if needed)
supabase init

# Create the function
supabase functions new multitenantkit
```

### 2. Write the Function

Edit `supabase/functions/multitenantkit/index.ts`:

```typescript
// Deno imports - IMPORTANT: import createClient directly from esm.sh
import { z } from 'https://deno.land/x/zod@v3.22.4/mod.ts';
import { createClient } from 'https://esm.sh/@supabase/supabase-js@2';
import { createSupabaseEdgeHandler } from 'https://esm.sh/@multitenantkit/sdk-supabase';

// Define custom fields (optional)
const customUserFields = z.object({
  firstName: z.string().min(1).max(100),
  lastName: z.string().min(1).max(100),
});

const customOrgFields = z.object({
  organizationName: z.string().min(1).max(200),
  industry: z.string().optional(),
});

// Create Supabase client - MUST use createClient from esm.sh directly
const client = createClient(
  Deno.env.get('PROJECT_URL')!,
  Deno.env.get('SERVICE_ROLE_KEY')!
);

// Create handler with custom fields
const handler = createSupabaseEdgeHandler({
  client,
  toolkitOptions: {
    users: {
      customFields: { customSchema: customUserFields }
    },
    organizations: {
      customFields: { customSchema: customOrgFields }
    }
  },
  edgeOptions: {
    basePath: '/multitenantkit'
  }
});

Deno.serve(handler);
```

<Callout type="warning">
**Important**: Always import `createClient` directly from `esm.sh/@supabase/supabase-js@2` and pass it to `createSupabaseEdgeHandler`. This avoids JWT verification failures caused by module version conflicts.
</Callout>

### 3. Deploy

```bash
supabase functions deploy multitenantkit
```

### 4. Test

Your API is now live:

```bash
# Base URL
https://your-project.supabase.co/functions/v1/multitenantkit

# Example: Get current user
curl https://your-project.supabase.co/functions/v1/multitenantkit/users/me \
  -H "Authorization: Bearer YOUR_SUPABASE_JWT"
```

## Test Your API

Both deployments expose the same 18 endpoints. Here are the key ones:

### Get Current User

```bash
curl -X GET http://localhost:3000/users/me \
  -H "Authorization: Bearer YOUR_JWT_TOKEN"
```

### Create Organization

```bash
curl -X POST http://localhost:3000/organizations \
  -H "Authorization: Bearer YOUR_JWT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"organizationName": "My Company"}'
```

### List User Organizations

```bash
curl -X GET http://localhost:3000/users/me/organizations \
  -H "Authorization: Bearer YOUR_JWT_TOKEN"
```

### Add Member to Organization

```bash
curl -X POST http://localhost:3000/organizations/{orgId}/members \
  -H "Authorization: Bearer YOUR_JWT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"username": "user@example.com"}'
```

## Available Endpoints

All 18 endpoints work identically in both deployment modes:

**Users (5)**
- `GET /users/me` - Get current user
- `PATCH /users/me` - Update current user
- `DELETE /users/me` - Delete current user
- `GET /users/me/organizations` - List user's organizations
- `POST /users` - Create user (called automatically on first auth)

**Organizations (8)**
- `POST /organizations` - Create organization
- `GET /organizations/:id` - Get organization
- `PATCH /organizations/:id` - Update organization
- `DELETE /organizations/:id` - Delete organization
- `POST /organizations/:id/archive` - Archive organization
- `POST /organizations/:id/restore` - Restore organization
- `POST /organizations/:id/transfer-ownership` - Transfer ownership
- `GET /organizations/:id/members` - List members

**Memberships (5)**
- `POST /organizations/:id/members` - Add member
- `POST /organizations/:id/accept` - Accept invitation
- `PUT /organizations/:id/members/:userId/role` - Update member role
- `DELETE /organizations/:id/members/:userId` - Remove member
- `DELETE /organizations/:id/members/me` - Leave organization
